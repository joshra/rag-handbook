# 01-名詞表與縮寫

> 目標：讓工程團隊在討論時「同一個詞只代表一件事」。

## 核心名詞

- **RAG（Retrieval-Augmented Generation）**：先「檢索」找證據，再「生成」組合答案的做法。
- **Retriever（檢索器）**：給定 query，回傳候選文件片段（chunks）的元件。
- **Chunk（片段）**：把原始文件切分後的最小檢索單位。通常會包含文字與 metadata。
- **Embedding（向量化）**：把文字映射成向量（高維數值陣列），用於相似度檢索。
- **Vector store / Vector DB（向量庫）**：保存向量與 metadata，支援相似度查詢。
- **pgvector**：PostgreSQL 的向量型別與相似度索引擴充，常用於「先用 DB 也能做」的落地。
- **BM25**：傳統關鍵字檢索/全文檢索的一種排名函數（常見於 Elasticsearch / OpenSearch）。
- **Hybrid retrieval（混合檢索）**：關鍵字檢索 + 向量檢索合併召回。
- **Rerank（重排）**：第二階段模型/規則，對候選 chunks 重新排序，提高最終相關性。
- **Context（上下文）**：提供給 LLM 的證據文本（通常是 top-k chunks 的整理/壓縮/引用）。
- **Grounding（依據化）**：要求生成答案必須根據提供證據，並能引用來源。

## 常見縮寫

- **top-k**：取相似度最高的 k 個候選 chunk。
- **MMR（Maximal Marginal Relevance）**：在相關性與多樣性間折衷的選取策略，避免 top-k 太相似。
- **ACL（Access Control List）**：存取控制清單；常落在 metadata filter。
- **PII**：個人可識別資訊。
- **SLA/SLO**：服務可用性/延遲等目標。
- **PoC**：概念驗證。

## 常見誤解（先寫在這裡避免踩雷）

- **「把所有文件丟進向量庫就會準」**：錯。資料清理、切分、metadata、檢索策略、rerank、prompt 才是關鍵。
- **「RAG 可以完全消除幻覺」**：錯。RAG 只能降低幻覺機率；仍需設計可回答/不可回答與引用規則。
- **「向量檢索一定比關鍵字檢索好」**：不一定。精確術語、代碼/型號、條款編號常常關鍵字更強；混合檢索通常更穩。

## 本章小結

- 名詞一致性是協作速度的倍增器：同一個詞只代表一件事。
- RAG 的準確度主要由資料清理、切分、metadata、檢索策略與引用規則共同決定。
- 混合檢索與 rerank 是上線穩定度的常見分水嶺。

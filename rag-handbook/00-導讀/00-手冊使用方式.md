# 00-手冊使用方式

## 本手冊定位

本手冊面向：希望把 RAG 導入既有系統（**Python + Django + PostgreSQL**）的工程團隊。

- **不綁定 LLM 供應商**：會用「抽象介面」描述 LLM / Embedding / Rerank 的互換點
- **以中文文件為主要情境**：切分、檢索、評估會特別提到中文常見問題
- **重視上線工程**：權限、審計、版本化、觀測與成本會放在主線，而不是附錄

## 本章地圖

- **適合用在**：你要帶團隊導入 RAG，需要先對準共同語言、閱讀路徑與驗收標準。
- **你會做出**：一條「從 0 到上線」的閱讀順序，以及可交付工件清單。
- **最可能踩雷**：只追求 demo；忽略權限/觀測/版本化，試點後變成不可維運的黑盒。

## 本書慣例（Conventions）

本手冊用固定的章節節奏與側欄（admonitions），讓讀者快速抓到「起手式」與「上線風險」：

> [!TIP]
> **起手式**：可以直接採用的預設流程/參數（先做對，再做大）。

> [!WARNING]
> **上線風險**：越權、資料外洩、成本失控、尾延遲暴增、不可回放。

> [!CAUTION]
> **常見誤解**：很容易做錯，而且做錯後修復成本高。

## 你需要先具備什麼

- **Python 後端基礎**：API、資料庫、背景任務（Celery/RQ/內建任務皆可）
- **Django 基礎**：middleware、permissions、model/migration、部署概念
- **PostgreSQL 基礎**：索引、查詢、交易與效能觀念

## 本手冊的「可交付」標準

你在實作時，至少能交付以下 4 個工件：

- **索引管線**：從文件來源 → 清理 → 切分 → embedding → 入庫（含版本化）
- **檢索 API**：支援 filter（權限/租戶/文件類型）與可觀測 log
- **生成策略**：有引用、可回答/不可回答策略、提示注入基本防護
- **評估與迭代**：至少一套離線集與「失敗分類」迴圈

## 本章小結

- 先建立共同語言（名詞表）與成功標準（KPI/可交付工件），再進入實作細節。
- 索引與重任務一律走背景任務；線上 query 以延遲與可回放為優先。
- 全程保持供應商不綁定：介面抽象優先於框架語法。

# 01-導入路線圖（PoC → 試點 → 正式）

## 你將學到（Learning Objectives）

- 能用最小風險導入：先證明價值，再擴大範圍
- 能把每階段的交付物定義清楚（避免一直卡在 PoC）
- 能把工程工作分成：資料治理、檢索品質、上線工程、產品體驗

## 本章地圖

- **適合用在**：要規劃導入節奏，避免 PoC 永遠卡在 PoC 時。
- **你會做出**：PoC→試點→正式的交付物、風險與驗收節點。
- **最可能踩雷**：只追求 demo；忽略資料權限與觀測，試點就爆炸。

## PoC（概念驗證）

目標：
- 證明「RAG 對你們場景有價值」：能回答一批代表性問題

交付物：
- 小型資料集（50～500 文件）
- 最小索引管線（可重跑）
- Query API（可回傳引用）
- 最小離線評估集（50～200 題）

成功門檻（示例）：
- recall@k 達到基本可用
- 有引用的回答可被團隊接受

## 試點（Pilot）

目標：
- 讓一個真實團隊/部門在真實流程中使用（帶回饋）

交付物：
- 權限/租戶隔離（硬性）
- 可觀測性（trace 回放）
- 回饋循環（負評→標註→改進）
- 成本/延遲基準（P95、每次成本）

## 正式（Production）

目標：
- 穩定、可擴展、可維運

交付物：
- 版本化（資料/索引/embedding/prompt）
- 降級策略（rerank/LLM/向量庫異常時）
- 安全合規（PII、提示注入、審計）
- SLO/SLA 與監控告警

## 常見導入失敗原因（反向清單）

- 沒有資料 owner → 索引很快變陳舊
- 沒有權限 filter → 不敢上線或上線後出事
- 沒有評估與觀測 → 無法迭代，只能「換模型賭運氣」

## 本章小結

- PoC 的目標是驗證價值與可行性；試點的目標是驗證可維運；正式才談規模化。
- 每一階段都要交付可回放與版本化能力，否則你無法自信地迭代。
- 把風險（越權/成本/延遲/品質）當成里程碑守門條件，而不是事後補洞。
